OpenAI が公開したコーディングエージェントのオーケストレーション仕様「Symphony」についてまとめました。

---

## Symphony とは

Symphony は OpenAI が 2026 年 4 月にオープンソース公開した、**コーディングエージェントのオーケストレーション仕様**です。

Linear などのプロジェクト管理ボードを「コーディングエージェントのコントロールプレーン」に変えることを目的としています。タスクボードを常時監視し、各タスクに自律エージェント（Codex）を割り当てて継続実行させます。エンジニアはエージェントを監視するのではなく、**タスクを管理するだけ**でよくなります。

## 仕組み

```
Linear ボード（タスク一覧）
       ↓ 30秒ごとにポーリング
  Symphony オーケストレーター
       ↓ タスクごとに割り当て
  Codex エージェント（隔離ワークスペースで実行）
       ↓ 完了したら
  PR を作成 → エンジニアがレビュー
```

- タスクが追加されると自動で拾い上げてエージェントを起動
- エージェントがクラッシュ・停止したら自動でリスタート（指数バックオフ）
- 1 タスク最大 20 ターンのループ実行
- `WORKFLOW.md` で動作ポリシーをチームごとにカスタマイズ可

## 6 つのアーキテクチャレイヤー

SPEC では実装が分離すべき 6 つの論理レイヤーが定義されています。

| レイヤー | 役割 |
|---|---|
| **ポリシー** | `WORKFLOW.md` でチームルールを定義 |
| **構成** | 設定の解析・デフォルト値適用 |
| **調整** | ポーリング・実行管理・リトライ（オーケストレーター本体） |
| **実行** | ワークスペース作成・エージェント起動 |
| **統合** | Linear API との接続・データ正規化 |
| **可視性** | 構造化ログ・ステータス表示 |

オーケストレーターだけがスケジューリング状態を変更でき、ワーカーはその結果を報告して状態遷移を起こします。

## 参照実装はElixirで書かれている

参照実装には Elixir が採用されています。並行プロセスの監視・オーケストレーションに優れたプリミティブを持つ言語として選ばれたとのことで、Codex がワンショットで実装したそうです。

なお、OpenAI は Symphony をスタンドアロン製品としてメンテしていく予定はなく、あくまで Codex App Server の活用例を示す「リファレンス実装」として位置付けています。チームがフォークして自分たちのユースケースに合わせて使うことを想定しています。

## 効果

導入した OpenAI 内のチームでは、最初の 3 週間でマージされた PR 数が **500% 増加**したと報告されています。

## 類似ツールとの比較

Symphony と似たコンセプトのツールはすでにいくつか存在しています。

| ツール | 特徴 |
|---|---|
| **Composio AO** | GitHub Issues / Linear / Jira に対応。CI エラー修正・レビューコメント対応まで自律で行う。プラグインで各コンポーネントを差し替え可能 |
| **T3 Code** | 同様のコーディングエージェントオーケストレーター。Claude Code アダプターも開発中 |
| **Claude Flow (Ruflo)** | Claude Code 上で 60 以上のエージェントをスウォーム実行。共有メモリ・RAG あり |
| **Kiro** | 実装前に要件定義書・設計書・データフロー図を自動生成するスペック駆動フロー |

Symphony は Linear 専用ですが、競合ツールは複数のトラッカーに対応しているものが多く、CI 連携やレビューコメントへの自動対応など、より自律度の高いものもあります。

## まとめ

Symphony は AIコーディングエージェントの「常時稼働オーケストレーター」であり、タスクボードをそのままエージェントへの指示書として使えるようにする仕様です。プロダクトというよりも、思想と仕組みを共有するためのリファレンスとして公開されているのが特徴で、今後のエージェント活用のあり方を示すものとして注目しています。

エンジニアが「エージェントに何をやらせるか」だけを考えればよい世界に着実に近づいてきているなと感じます。
